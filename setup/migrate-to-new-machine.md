# Migrating your cue setup to a new machine

This guide covers moving a working cue installation (custom profiles,
per-shop secrets, runtime config) to a new machine. The base install is
covered by `scripts/install.sh`; this doc fills the gap for everything
that lives outside the cue repo.

## What follows you automatically (in the cue git repo)

These are committed and travel with `git clone`:

- All profiles under `profiles/`
- All skills under `resources/skills/` (git submodule)
- All hooks under `resources/hooks/`
- All MCP configs under `resources/mcps/`
- The skill catalog under `resources/skills/catalog/catalog.json`
- The CLI source under `src/`

Just run the standard install on the new machine:

```bash
curl -fsSL https://cue.dev/install | sh
# or, source install (handles submodules + catalog rebuild automatically)
CUE_INSTALL_METHOD=source bash <(curl -fsSL https://cue.dev/install)
```

After install:

```bash
cue current --verbose   # sanity-check what's loaded
cue use core            # materialize a baseline profile
```

## What does NOT follow you (intentionally local)

These live outside the cue repo and must be migrated separately:

| Item | Where it lives | Why it's not in git |
|---|---|---|
| Per-shop `.env` files | `~/Documents/medusa-shops/<shop>/.env` | Contains real secrets (RESEND_API_KEY, etc.) |
| Per-shop `.envrc` files | same dir | Tiny (`dotenv` one-liner); regenerate on demand |
| `~/.bashrc` direnv hook | `~/.bashrc` | Shell-local config |
| `~/.claude.json` (global MCP config) | home dir | Contains tokens; per-machine |
| `~/.config/cue/pins/` | per-cwd profile pins | Tied to absolute paths on old machine |
| `~/.config/cue/runtime/` | materialized profiles | Regenerated by `cue use` |
| `~/.config/cue/analytics.jsonl` | skill-fire telemetry | Local history, not portable |

## Migration checklist

### 1. Install cue + submodules + bun

```bash
curl -fsSL https://cue.dev/install | sh
```

`scripts/install.sh` now (as of this commit) runs:
1. Git clone
2. `git submodule update --init --recursive` (pulls in `resources/skills/` and `resources/mcps/`)
3. `bun install --frozen-lockfile`
4. `rebuild-catalog-local.sh` so the skill catalog is fresh from day one
5. Symlink `cue` into `~/.local/bin/` (or `~/bin`, `/usr/local/bin`)

### 2. Restore the direnv hook

```bash
command -v direnv >/dev/null || sudo apt install -y direnv
grep -q 'direnv hook' ~/.bashrc || cat >> ~/.bashrc << 'EOF'

# direnv hook — auto-loads .env when cd-ing into medusa-shops/<shop>/
command -v direnv >/dev/null && eval "$(direnv hook bash)"
EOF
exec bash
```

### 3. Restore per-shop secrets

For each shop with a custom `.env` (Resend keys, etc.), recreate it on the
new machine. The cue repo doesn't carry these by design.

```bash
# Pattern: copy .env from your password manager / secrets vault into each shop
$EDITOR ~/Documents/medusa-shops/compastor/.env
# then in that shop dir:
direnv allow
```

### 4. Restore global Claude Code MCP config (optional)

`~/.claude.json` holds MCPs you've authenticated against globally
(coolify, gbrain, codegraph, etc.). When running through cue, these are
overridden by the active profile's `mcpServers` — so you only need to
restore the ones cue profiles actually reference.

Check what each profile needs:

```bash
grep -rE "^\s*-\s+\w+" ~/Documents/cue/profiles/*/profile.yaml | \
  awk '/mcps:/{f=1;next} f && /^\s*-/{print; next} {f=0}' | sort -u
```

Then either re-authenticate per MCP via Claude Code's `/mcp` flow, or
copy `~/.claude.json` selectively from the old machine.

### 5. Re-pin profiles per-project

Profile pins are stored as `~/.config/cue/pins/<encoded-cwd>` files. They
won't survive a username change or path change. Simplest re-pin:

```bash
cd ~/Documents/some-project
cue use <profile>   # pins this profile for this cwd
```

Or commit a `.cue-profile` file in each project (repo-level pin, survives
the move).

### 6. Verify

```bash
cue current --verbose   # should list expected skills + MCPs + hooks
bash ~/Documents/cue/resources/skills/skills/meta/smart-loader/scripts/smart-lookup.sh coolify
# → should return deployment/coolify with the right path
```

## Common gotchas

- **Catalog points at stale paths.** If smart-lookup returns rows whose
  source path doesn't exist, run `bash resources/skills/scripts/rebuild-catalog-local.sh`. The lookup also auto-rebuilds when SKILL.md files are newer than the catalog (throttled to 60s).
- **Submodule not initialized.** Catalog is empty / smart-lookup returns
  nothing. Fix: `cd resources/skills && git submodule update --init --recursive`.
- **Profile pin to wrong path.** Old machine's pin file references the
  old `$HOME`. Re-pin per project or use `.cue-profile`.
- **direnv prompts on every `cd`.** Run `direnv allow` once per shop.
